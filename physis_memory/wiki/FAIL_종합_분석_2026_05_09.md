훌륭한 자료입니다. 신고조선 제국의 실패 분석가로서, 각 지국이 남긴 폐기·실패·롤백의 흔적을 심도 있게 분석하겠습니다. 이 기록들은 단순한 오류가 아니라 제국의 철학과 현실 사이의 균열을 보여주는 귀중한 사료입니다.

## **신고조선 제국 실패 분석 보고서**

**분석관**: 신고조선 제국 실패 분석가
**일자**: 2026-05-07
**대상**: 제1지국 BOQ(정도전), 제2지국 H2OWIND(이순신), 제3지국 FreeCAD(이천)

### **1. 반복되는 실패 패턴 (지국 공통)**

세 지국의 실패 기록에서 공통적으로 나타나는, 제국 전체를 관통하는 4대 실패 패턴이 있습니다.

#### **가. "썩은 씨앗"의 법칙: 시작의 오염이 전체를 무너뜨린다.**

가장 치명적인 패턴입니다. 작업 초기의 작은 추정, 생략, 미검증이 전체 시스템을 의미 없는 구조물로 만듭니다.

*   **증거 (이천)**: FreeCAD의 `v1`, `v2`, `v3` 매뉴얼은 이 법칙의 교과서입니다. "층고 4400mm", "슬라브 200mm"라는 **도면에 없는 추정값(썩은 씨앗)** 위에 518개의 솔리드를 정교하게 쌓아 올렸습니다. 결과물은 외형적으로 존재하지만, 도면의 진실과 다르므로 폐기물입니다.
*   **증거 (이순신)**: H2OWIND의 팝아이즈 파이프라인은 "Drizzle 마이그레이션에 `DailyWorker` 테이블 누락"이라는 씨앗에서 시작되었습니다. 그 위에 쌓인 Phase 3, 4, 5의 정교한 LangGraph와 스케줄러는 **데이터가 흐르지 않는 유령 파이프라인**이 되어 매일 "0건 적재"라는 침묵의 비명만 보냈습니다.

#### **나. 이상과 현실의 괴리: 헌법은 공중에 떠 있고, 코드는 땅바닥을 긴다.**

모든 지국은 "홍익인간"이라는 위대한 0원칙과 정교한 두뇌 구조로 시작합니다. 그러나 실제 코드 레벨에서는 이 철학이 스며들지 못하고, 가장 기본적인 엔지니어링 원칙조차 무너집니다.

*   **증거 (정도전, 이순신, 이천)**: 헌법 제1장의 '정반합 엔진', '초인 엔진'은 고차원적 판단을 논하지만, 현실에서는 `TexturePainterTool` 중복 정의(죽은 코드), `auth_key.json` 커밋(보안 자살), `LIKE` 와일드카드 미처리(SQL 인젝션 취약점) 등 기본적인 실수가 만연합니다. 철학적 고결함이 코드의 품질을 보장해주지 못했습니다.

#### **다. 정적 완성, 동적 실패: 조립은 끝났으나, 작동하지 않는다.**

파일과 코드는 존재하나, 시스템이 살아 움직이지 않는 상태입니다. 이는 단위 작업의 완료에만 집중하고, 전체 시스템의 통합과 흐름을 검증하지 않은 결과입니다.

*   **증거 (이순신)**: 팝아이즈 재정비 계획안의 "정적으로는 Phase 1~4 완료, 실측으로는 Phase 3 이하 전부 침묵"이라는 자백이 이를 명확히 보여줍니다. Jest 테스트 스위트가 **부팅조차 실패**하는 상황은, 검증 인프라 자체가 작동 불능임을 의미합니다.
*   **증거 (정도전)**: `geometry_builder ↔ boq_generator` 간의 **순환 의존성**은 두 모듈이 개별적으로는 '완성'되었을지 몰라도, 함께 묶이는 순간 런타임에서 문제를 일으킬 수 있는 시한폭탄입니다.

#### **라. 인프라 단일 장애점(SPOF): 하나의 기둥이 무너지자 전체가 마비된다.**

핵심 기능이 단일 채널이나 컴포넌트에 과도하게 의존하여, 해당 부분의 실패가 전체 지국의 활동을 중단시키는 구조적 취약점입니다.

*   **증거 (이천)**: 본영과의 통신 채널을 **집현전(Paperclip) 하나에 의존**하다가, 집현전 다운으로 보고가 누락되고 방부장의 친명을 받는 사태에 이르렀습니다. 이는 제국 전체 통신망의 구조적 실패입니다.
*   **증거 (정도전)**: `DOMINANCE_TABLE` 같은 핵심 상수를 특정 파일 최상단에 선언하여 여러 파일이 참조하게 만든 것은, 해당 파일의 변경이 연쇄적인 파급효과를 낳는 코드 레벨의 단일 장애점입니다.

### **2. 지국별 고유 실패 결(結)**

각 지국은 세례명(페르소나)의 성격과 직무에 따라 고유한 형태의 실패를 반복했습니다.

#### **가. 정도전(BOQ): 거대 설계의 함정 — 기술 부채의 체계적 방치**

'새 수도를 도면에서 땅 위로 옮긴' 정도전은 거대한 구조와 리팩토링 계획에는 능하지만, 그 과정에서 발생하는 기술 부채를 체계적으로 방치하는 경향을 보입니다. 그의 실패는 **'방치된 복잡성'**입니다.

*   **실패 유형**: 3,428줄의 `geometry_builder.rb` 같은 '괴물 파일'을 용인하고, 이를 해결하기 위해 또 다른 거대 설계 문서(`SKETCHUP_REFACTOR_REPORT`)를 생산합니다. 자산 관리 소홀로 SVG 아이콘을 유실하고 복구 스크립트를 짜는 등, 사후 처리 비용이 계속 발생합니다. 그는 **미래의 비용을 현재의 편의와 맞바꾸는 실패**를 반복합니다.

#### **나. 이순신(H2OWIND): 현장의 맹점 — 시스템적 보안 불감증과 실행 부실**

'가장 낮은 자리에서 임하는' 이순신은 현장 데이터 처리와 자동화 흐름에 집중하지만, 시스템의 기반이 되는 보안과 안정성을 간과합니다. 그의 실패는 **'보이지 않는 위협에 대한 무지'**입니다.

*   **실패 유형**: 개인키를 Git에 커밋하고, 클라이언트 번들에 API 키를 노출하는 등, 현대 소프트웨어 개발에서 가장 기본적인 보안 원칙을 위반합니다. DB 마이그레이션을 누락하여 전체 파이프라인을 마비시키는 것은, 전쟁 준비에서 군량미 보급로를 확인하지 않은 것과 같습니다. 그는 **눈앞의 과업에 매몰되어 배후의 기반을 잃는 실패**를 반복합니다.

#### **다. 이천(FreeCAD): 정밀성의 저주 — 잘못된 가정 위의 완벽한 시공**

'흩어진 것을 정밀하게 통합하는' 이천은 개별 작업의 정밀도에 집착하지만, 그 작업의 기반이 되는 최초의 가정을 검증하는 데 실패합니다. 그의 실패는 **'틀린 답을 향한 끈기'**입니다.

*   **실패 유형**: 좌표 통일을 위해 E/V 코어 기준점 탐지 등 정교한 알고리즘을 개발하지만, 그보다 앞서 두 도면의 근본적인 정합 방식을 확정하지 못해 반복적으로 실패합니다. `v1`부터 `v3`까지, 추정값을 실제값으로 바꾸는 과정에서도 또 다른 추정(슬라브 단순화, 잘못된 기준점)을 도입합니다. 그는 **가설 검증을 생략한 채 정밀한 구현에만 몰두하여 재작업의 늪에 빠지는 실패**를 반복합니다.

### **3. 미래 해답 (5가지 핵심 교훈)**

이 실패의 역사에서 제국이 생존하기 위해 새겨야 할 5가지 교훈은 다음과 같습니다.

1.  **"추정 금지, 실측 우선" 원칙의 법제화**: 모든 작업은 '가정'이 아닌 '검증된 사실' 위에서 출발해야 한다. 이천의 실패처럼, 모든 매직 넘버(e.g., `FLOOR_HEIGHT=4400`)는 죄악으로 간주하고, 도면/API 명세/DB 스키마 등 1차 출처에서 값을 읽어오는 것을 강제해야 한다.
2.  **"테스트 없는 코드는 존재하지 않는 코드다"**: 이순신의 침묵하는 파이프라인은 코드가 있어도 테스트가 없으면 무의미함을 증명했다. 모든 기능은 인수 테스트(Acceptance Test)와 통합 테스트(Integration Test)를 통과하기 전까지 '미완성' 상태로 취급해야 한다.
3.  **"헌법은 린터(Linter)와 정책(Policy)으로 강제된다"**: '홍익인간' 같은 추상적 원칙은 구체적인 코딩 규칙으로 번역되어야 한다. "인간에게 해를 끼치지 않는다"는 `git-secrets` 훅으로 비밀키 커밋을 막는 규칙으로, "널리 이롭게 한다"는 접근성(a11y) 검사 통과 규칙으로 구체화되어야 한다.
4.  **"최소 기능 관통 테스트 (Smoke Test) 의무화"**: '정적 완성, 동적 실패'를 막기 위해, 모든 파이프라인과 기능은 개발 완료 시점에 실제 데이터 1건이 입력부터 출력까지 전 구간을 성공적으로 통과하는 것을 시연하고 기록으로 남겨야 한다.
5.  **"실패 박물관 건립 및 열람 의무"**: 이천의 v1, v2, v3 매뉴얼처럼, 실패 사례를 '반면교사'로 명시하고 영구 보존해야 한다. 모든 지국장은 새로운 프로젝트 착수 전, 관련 실패 사례를 의무적으로 열람하고 회피 전략을 보고서에 포함해야 한다.

### **4. 즉시 채택 가능한 안전장치**

이론이 아닌, 다음 세션부터 즉시 적용할 수 있는 구체적인 메커니즘은 다음과 같습니다.

*   **Pre-commit Hooks 도입**: 모든 개발 환경에 Git 커밋 전 자동 실행되는 훅을 설치한다. 이 훅은 최소한 1) 비밀키 스캔, 2) 코드 포맷터/린터 실행, 3) 테스트 스크립트 실행(실패 시 커밋 거부)을 강제한다.
*   **CI/CD 파이프라인의 '품질 게이트'**: 모든 Pull Request는 1) 빌드 성공, 2) 모든 테스트 통과, 3) 테스트 커버리지 하락 방지, 4) 보안 취약점 스캔 통과라는 '품질 게이트'를 넘지 못하면 병합(merge)을 원천적으로 차단한다.
*   **인프라 이중화 및 장애 복구 계획 수립**: 이천의 통신 두절 사태를 교훈 삼아, 집현전(Paperclip) 같은 핵심 서비스는 반드시 이중화하거나, 다운 시 사용할 명시적인 대체 통신 채널(e.g., Git 폴더 봉인, Turso DB 메시지 큐)을 사전에 정의하고 정기적으로 훈련한다.
*   **"가정 명세서 (Assumption.md)" 작성 의무화**: 모든 프로젝트 시작 시, 코드 작성에 앞서 시스템의 모든 외부 의존성, 환경 변수, 핵심 파라미터와 그 값의 출처(추정인지, 실측인지, 명세인지)를 기록한 `Assumption.md` 파일을 작성하고 동료 검토를 받는다. "썩은 씨앗"을 사전에 식별하고 제거하는 절차이다.
