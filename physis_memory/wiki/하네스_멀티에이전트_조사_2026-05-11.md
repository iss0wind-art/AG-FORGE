### 섹션 1: 에이전트 하네스 (Agent Harness)

#### 1. 하네스의 정의
**핵심 정의:** 에이전트 하네스(Agent Harness)란, 거대 언어 모델(LLM)이라는 '엔진'을 실제 세상에서 유의미한 작업을 수행하는 '자율 에이전트'로 구동시키기 위한 실행 환경이자 구조체(Scaffolding)입니다. 이는 단순히 LLM을 API로 호출하는 것을 넘어, ▲도구(Tools) ▲메모리(Memory) ▲실행 루프(Loop) ▲정책(Policy)과 같은 핵심 구성요소들을 LLM 본체와 유기적으로 묶어, 목표 지향적인 행동을 지속적으로 이끌어내는 구동체를 의미합니다.

자동차의 '와이어링 하네스'가 엔진의 동력을 각종 전장 부품에 전달하고 제어하듯, 에이전트 하네스는 LLM의 추론 능력을 파일 시스템, API, 데이터베이스 등 외부 세계와 연결하고 그 상호작용을 통제합니다.

**실무 함의:**
*   **사실:** LLM 모델 자체는 텍스트를 생성할 뿐, 파일을 직접 수정하거나 API를 호출할 수 없습니다.
*   **추론:** 따라서 에이전트의 실제 성능은 LLM 모델의 지능만큼이나, 이 지능을 얼마나 효과적으로 실행에 옮기게 하는 하네스의 설계에 따라 결정됩니다. 좋은 하네스는 평범한 모델의 성능을 극대화하고, 나쁜 하네스는 최고의 모델조차 무력하게 만듭니다.

#### 2. 2025~2026년 유행 배경
2024년 Devin의 등장과 Anthropic의 Claude Code, Microsoft의 OpenHands, GitHub Copilot Workspace 등의 코딩 에이전트 경쟁이 격화되면서, 업계는 중요한 사실을 깨달았습니다.

*   **사실 (배경):** 초기 코딩 에이전트들은 단발성 코드 생성에 그쳤으나, Devin, OpenHands 등은 스스로 파일을 생성하고, 명령어를 실행하며, 테스트를 돌리고, 에러를 디버깅하는 등 복잡한 소프트웨어 엔지니어링 '프로세스'를 수행하려 시도했습니다. 이 과정에서 수많은 시행착오와 상태 관리가 필수적이었습니다.
*   **추론 (유행 이유):** 이러한 시도들은 LLM의 '순수 지능'만으로는 부족하다는 것을 명백히 보여주었습니다. 성공적인 에이전트는 모델이 내놓은 계획(plan)을 안정적으로 실행(act)하고, 그 결과를 관찰(observe)하여 다음 계획에 반영하는 체계적인 '운영체제(OS)'를 갖추고 있었습니다. 이 운영체제의 역할이 바로 '하네스'이며, 에이전트의 성능을 결정짓는 핵심 변수임이 드러나자, 하네스 설계와 추상화는 더 이상 부수적인 '래퍼(wrapper)'가 아닌, 에이전트 공학의 **1급 시민(First-class Citizen)**으로 격상되었습니다. 모델이 두뇌라면, 하네스는 그 두뇌를 담는 두개골이자 온몸으로 이어지는 신경계와 같은 존재로 인식되기 시작한 것입니다.

#### 3. 핵심 구성요소
1.  **시스템 프롬프트 (System Prompt):** 에이전트의 정체성, 목표, 행동 원칙, 제약 조건을 정의하는 '헌법'입니다. 어떤 도구를 사용할 수 있는지, 어떤 스타일로 응답해야 하는지, 실패 시 어떻게 행동해야 하는지를 규정합니다.
2.  **도구 스키마 (Tool Schema):** 에이전트가 사용할 수 있는 도구(함수, API 등)의 명세서입니다. 보통 OpenAPI Specification이나 JSON Schema 형태로 정의되며, LLM이 각 도구의 이름, 인자, 반환값을 명확히 이해하고 올바른 형태로 호출(Function Calling)할 수 있게 돕습니다.
3.  **컨텍스트 관리 (Context Management):** LLM의 제한된 컨텍스트 창(Context Window) 안에서 장기적인 작업을 수행하기 위한 기술입니다.
    *   **요약/압축:** 대화가 길어지거나 도구 실행 결과가 방대할 때, 핵심 정보를 요약하여 컨텍스트 부담을 줄입니다.
    *   **RAG (Retrieval-Augmented Generation):** 관련 정보를 외부 벡터 DB 등에서 실시간으로 가져와 컨텍스트에 주입합니다.
4.  **실행 루프 (Execution Loop):** 에이전트가 작업을 반복적으로 수행하는 핵심 엔진입니다. 가장 대표적인 패턴은 **ReAct (Reason + Act)**로, `계획(plan) -> 행동(act) -> 관찰(observe)` 사이클을 반복하며 목표에 점진적으로 다가갑니다.
5.  **안전장치 (Safety Guards):** 에이전트의 자율적인 행동이 초래할 위험을 통제하는 장치입니다.
    *   **승인 (Approval):** 파일 삭제, 코드 배포 등 위험한 행동 전 사용자에게 승인을 요청합니다.
    *   **차단기 (Circuit Breaker):** 정해진 횟수 이상의 루프, 과도한 비용 발생, 특정 금지 행위 시도 시 에이전트 작동을 강제로 중단시킵니다.
6.  **관측성 (Observability):** 에이전트의 내부 동작을 추적하고 디버깅하기 위한 장치입니다.
    *   **트레이스 (Trace):** 에이전트의 생각(LLM 추론 과정), 도구 호출, 결과 등 모든 단계를 시각적으로 추적할 수 있게 합니다. (예: LangSmith)
    *   **로그 (Log):** 구조화된 로그를 통해 에이전트의 상태 변화와 의사결정 과정을 기록합니다.

#### 4. 대표 오픈/상용 하네스
*   **사실 (특징 비교):**
    *   **Claude Agent SDK / OpenAI Assistants API:** 상용 플랫폼 종속적인 하네스입니다. 도구 사용, 상태 관리 등을 API 뒤로 추상화하여 개발 편의성이 높지만, 내부 실행 루프나 컨텍스트 관리의 세밀한 제어는 어렵습니다. 2026년 현재, 대부분의 간단한 에이전트는 이들을 기반으로 구축됩니다.
    *   **OpenHands (구 AllHands), SWE-agent:** 특정 벤치마크(SWE-bench) 해결에 특화된 오픈소스 하네스입니다. 파일 시스템 조작, 셸 명령어 실행 등 소프트웨어 엔지니어링 작업에 필요한 강력한 도구와 정교한 실행 루프를 갖추고 있습니다. 연구 목적과 고난도 작업에 적합합니다.
    *   **Aider, smol-developer:** 개발자의 IDE 환경에 통합되어, 인간과 긴밀하게 협력하는 '페어 프로그래머' 역할에 집중합니다. 전체 프로젝트를 자율적으로 구축하기보다는, 명시적인 지시에 따라 코드베이스와 상호작용하는 데 특화된 경량 하네스입니다.
    *   **AutoGPT 계열:** 에이전트 개념의 대중화를 이끈 선구자이지만, 2026년 기준으로는 개념 증명(PoC) 성격이 강합니다. 명확한 상태 관리나 안정적인 실행 루프가 부족하여 실제 프로덕션 환경에 적용하기에는 신뢰성이 낮다는 평가가 지배적입니다.

*   **추론 (차이의 본질):** 이들의 핵심 차이는 **'자율성의 수준'**과 **'작업의 범위'**에 있습니다. Assistants API는 제한된 자율성을 부여하는 범용 하네스이며, OpenHands는 완전 자율성을 지향하는 전문 하네스, Aider는 인간의 통제 하에 작동하는 협업 하네스라고 볼 수 있습니다.

#### 5. 하네스 설계 안티패턴
1.  **무한 루프 (Infinite Loops):** 에이전트가 목표 달성에 실패하고 동일한 행동을 무의미하게 반복하는 상태. 명확한 종료 조건이나 상태 변화 감지 로직이 부재할 때 발생합니다.
2.  **도구 폭주 (Tool Spamming):** 잘못된 도구를 반복적으로 호출하거나, 같은 도구를 부적절한 인자로 계속 호출하는 현상. 도구 스키마 설명이 불분명하거나 LLM이 상황을 오해할 때 발생합니다.
3.  **컨텍스트 오염 (Context Pollution):** 도구 실행 결과의 장황한 에러 메시지나 불필요한 정보가 컨텍스트 창을 가득 채워, LLM이 다음 행동을 위한 핵심 정보를 놓치게 만드는 문제입니다.
4.  **평가 부재 (Lack of Evaluation):** 특정 성공 사례(cherry-picking)에만 의존하고, 다양한 시나리오에 대한 체계적인 평가 없이 에이전트를 설계하는 것. 이는 실제 환경에서 쉽게 실패하는 '데모용 에이전트'를 낳습니다.

#### 6. 평가·벤치마크
*   **사실:** SWE-bench (실제 GitHub 이슈 해결), TerminalBench (터미널 명령어 수행), OSWorld (운영체제 수준의 복합 작업)와 같은 벤치마크가 등장했습니다. 이들은 에이전트가 실제 세계의 복잡한 문제를 얼마나 잘 해결하는지를 측정합니다.
*   **추론 (분리 평가의 의의):** 이러한 벤치마크의 가장 큰 의의는 **"모델 지능 vs. 하네스 효율성"**을 분리하여 평가할 수 있는 척도를 제공한다는 점입니다. 예를 들어, 같은 GPT-5 모델을 사용하더라도 A 하네스는 SWE-bench에서 30%의 성공률을, B 하네스는 50%의 성공률을 보일 수 있습니다. 이는 B 하네스의 실행 루프, 컨텍스트 관리, 오류 처리 능력이 더 우월함을 의미합니다. 이로써 우리는 막연히 "더 좋은 모델"만 기다리는 것이 아니라, 엔지니어링의 영역인 '하네스'를 개선함으로써 에이전트의 성능을 실질적으로 향상시킬 수 있다는 명확한 방향성을 갖게 되었습니다.

### 섹션 2: 멀티 에이전트 (Multi-Agent)

#### 1. 멀티 에이전트 시스템의 정의
**핵심 정의:** 멀티 에이전트 시스템(Multi-Agent System, MAS)이란, 단일 LLM 호출이나 단일 에이전트가 모든 것을 처리하는 방식이 아니라, 각기 다른 역할, 목표, 전문성을 가진 **다수의 자율 에이전트들이 서로 상호작용(협업, 경쟁, 협상)하며 공동의 목표를 달성하거나 복잡한 시스템을 시뮬레이션하는 구성**을 말합니다.

이는 마치 한 명의 천재 개발자가 모든 것을 다 하는 대신, 기획자, 개발자, QA 테스터, 프로젝트 관리자로 구성된 '팀'을 꾸려 문제를 해결하는 것과 같습니다.

**실무 함의:**
*   **사실:** 단일 에이전트는 복잡한 문제를 해결할 때 인지적 부담이 커지고, 하나의 관점에 갇히기 쉽습니다.
*   **추론:** 멀티 에이전트는 '역할 분담'을 통해 각 에이전트의 프롬프트와 도구를 단순화하고, '상호작용'을 통해 단일 에이전트가 놓칠 수 있는 오류를 발견하거나 더 창의적인 해결책을 도출할 잠재력을 가집니다.

#### 2. 주요 패턴
1.  **Orchestrator-Worker (조정자-작업자):** 중앙의 조정자(또는 라우터) 에이전트가 사용자 요청을 분석하여, 가장 적합한 전문 작업자 에이전트에게 작업을 할당하는 패턴입니다. 가장 흔하고 직관적인 구조입니다.
2.  **Debate/Critic (토론/비평가):** 하나의 에이전트가 초안(예: 코드, 보고서)을 생성하면, 다른 비평가 에이전트가 그 결과물을 검토하고 피드백을 제공합니다. 이 과정을 반복하며 결과물의 품질을 높입니다.
3.  **Hierarchical (계층적):** CEO 에이전트가 큰 목표를 설정하면, 중간 관리자 에이전트들이 이를 세부 과업으로 나누고, 실무자 에이전트들이 실제 실행을 담당하는 회사 조직도와 같은 구조입니다.
4.  **Swarm (군집):** 중앙 통제 없이 다수의 동등한 에이전트들이 각자의 규칙에 따라 독립적으로 행동하며, 그 상호작용을 통해 집단적으로 복잡한 문제를 해결하거나 창발적인 결과를 만들어내는 패턴입니다. (예: 아이디어 생성)
5.  **Blackboard (칠판):** 모든 에이전트가 접근할 수 있는 공유된 '작업 공간(칠판)'을 두고, 각자 기여할 수 있는 정보나 해결책을 칠판에 적으면 다른 에이전트들이 이를 참조하여 작업을 이어가는 협업 방식입니다.
6.  **Reflexion (반성):** 엄밀히는 단일 에이전트 기법이지만 멀티 에이전트 패턴과 유사하게 작동합니다. 에이전트가 자신의 행동 결과를 평가하고, 실패 원인을 분석하여 다음 시도에서 계획을 수정하는 '내면의 비평가'를 두는 방식입니다.

#### 3. 대표 프레임워크
*   **사실 (특징 비교):**
    *   **LangGraph:** 상태 기계(State Machine) 개념을 기반으로 에이전트 간의 상호작용을 '그래프'로 명시적으로 정의합니다. 순환(cycle)을 허용하여 토론이나 수정 같은 반복적인 협업 흐름을 구현하는 데 매우 강력하며, 안정성이 높습니다.
    *   **CrewAI:** 역할 기반(Role-based) 접근법을 사용하여, 각 에이전트의 역할(role), 목표(goal), 배경(backstory)을 쉽게 정의할 수 있도록 하는 고수준 프레임워크입니다. Orchestrator-Worker나 순차적 협업 패턴을 빠르게 구축하는 데 유리합니다.
    *   **AutoGen (Microsoft):** 에이전트 간의 '대화'를 중심으로 시스템을 설계합니다. 유연성이 매우 높아 복잡한 연구나 시뮬레이션에 적합하지만, 대화 흐름을 제어하기가 상대적으로 어렵습니다.
    *   **OpenAI Swarm / Anthropic Claude Agent SDK의 sub-agent / Microsoft Magentic-One:**
        *   **추론 (2026년 동향):** 이들은 2026년 현재 주요 AI 기업들이 자사 플랫폼에 내장하려는 차세대 멀티 에이전트 기능들을 대표하는 가상의 이름 또는 초기 버전입니다. 이들의 공통점은 단일 에이전트 하네스 위에서 여러 에이전트를 '자식 프로세스'처럼 생성하고 관리하는 기능을 네이티브로 제공하여, 개발자가 복잡한 통신 프로토콜 없이도 계층적 패턴 등을 쉽게 구현할 수 있도록 돕는다는 것입니다.

#### 4. 멀티 에이전트가 빛나는 영역 vs 단일 에이전트가 더 나은 영역
*   **멀티 에이전트가 빛나는 영역:**
    *   **사실 (실증 결과):** 복잡한 소프트웨어 개발(기획-코딩-테스트-배포), 과학적 연구(가설-실험-분석), 투자 분석 보고서 작성(데이터수집-분석-초안작성-검토) 등 명확한 역할 분담이 가능하고, 다양한 관점의 검토가 결과물의 품질을 극적으로 향상시키는 영역에서 효과를 보입니다.
    *   **추론:** 즉, 문제 자체가 '팀워크'를 요구하는 형태일 때 가장 강력합니다.
*   **단일 에이전트가 더 나은 영역:**
    *   **사실 (실증 결과):** "이메일 분류하기", "API 데이터 요약하기", "간단한 스크립트 작성하기" 등 작업의 절차가 선형적이고 명확한 대부분의 과업에서는 단일 에이전트가 훨씬 효율적입니다.
    *   **추론 (트레이드오프):** 멀티 에이전트는 필연적으로 에이전트 간 통신으로 인한 **비용(토큰 사용량) 증가**, **지연 시간(Latency) 증가**, **상태 동기화의 복잡성**, **디버깅 난이도 급증**이라는 대가를 치릅니다. 단일 에이전트로 충분히 해결 가능한 문제에 멀티 에이전트를 도입하는 것은 '닭 잡는 데 소 잡는 칼을 쓰는 격'입니다.

#### 5. 핵심 난제
1.  **통신 프로토콜 (Communication Protocol):** 에이전트들이 어떻게 정보를 교환할 것인가? 자연어 채팅? 구조화된 JSON? 통신의 효율성과 표현력 사이의 트레이드오프가 존재합니다.
2.  **메모리/상태 공유 (Shared Memory/State):** 'A 에이전트가 알게 된 사실'을 'B 에이전트'가 어떻게, 언제 알게 할 것인가? 공유 데이터베이스, 칠판 패턴 등을 사용하지만 동기화 문제가 발생하기 쉽습니다.
3.  **합의 알고리즘 (Consensus Algorithm):** 여러 에이전트의 의견이 다를 때, 최종 결정을 어떻게 내릴 것인가? 투표, 상급자(보스 에이전트)의 결정 등 다양한 방식이 필요합니다.
4.  **비용 폭증 (Cost Explosion):** 에이전트 N개가 서로 대화하면, LLM 호출 횟수는 N배 이상으로 기하급수적으로 늘어날 수 있습니다.
5.  **종료 조건 (Termination Condition):** 언제 이들의 '회의'나 '작업'이 끝났다고 판단할 것인가? 명확한 종료 조건이 없으면 무한한 토론에 빠질 수 있습니다.

#### 6. 2026년 동향
*   **추론 (주요 경향):** 2026년 현재, LLM 모델 자체의 추론 능력과 컨텍스트 처리 용량이 비약적으로 발전하면서, **"하나의 매우 강력한 모델 + 잘 설계된 하네스"** 조합이 어설픈 **"여러 개의 약한 모델 + 복잡한 멀티 에이전트"** 조합을 대부분의 영역에서 압도하는 경향이 뚜렷해지고 있습니다. 강력한 단일 모델은 내부적으로 '전문가 페르소나'를 시뮬레이션하여 멀티 에이전트의 장점을 일부 흉내 낼 수 있으며, 이는 비용과 속도 면에서 훨씬 효율적입니다.

*   **추론 (멀티 에이전트의 본질적 가치):** 그럼에도 불구하고 멀티 에이전트가 본질적으로 필요한 자리는 여전히 존재하며, 오히려 그 역할이 더 명확해지고 있습니다.
    1.  **진정한 이질성(Heterogeneity)이 필요할 때:** 서로 다른 기반 모델(예: Claude는 창의성, GPT는 코딩)이나 서로 다른 사설 데이터를 학습한 에이전트들을 조합하여 시너지를 내야 할 경우.
    2.  **견고한 고립(Isolation)이 필요할 때:** 보안을 위해 공격자 역할을 하는 '레드팀 에이전트'와 방어자 역할을 하는 '블루팀 에이전트'가 서로의 내부 상태를 절대 공유해서는 안 되는 사이버 보안 시뮬레이션.
    3.  **사회적 상호작용 자체가 목적인 경우:** 경제 모델 시뮬레이션, 게임 NPC 동작 생성 등 에이전트 간의 상호작용 자체가 분석 대상일 때.
